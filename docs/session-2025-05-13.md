## Bugs encontrados y corregidos

### ESP32 (`sensor_golpes.ino`)

#### 1. Advertising desaparece a los pocos segundos
- **Causa**: Algunas versiones del SDK de Espressif ponen un timeout implícito al advertising.
- **Fix**: Loop que llama `BLEDevice::startAdvertising()` cada 8s cuando no hay cliente conectado.

#### 2. BLE invisible ~5 segundos al arrancar
- **Causa**: `calibrar()` (3.5s) corría **antes** de `iniciarBLE()`.
- **Fix**: `iniciarBLE()` se movió al inicio de `setup()`, antes de `calibrar()`.

#### 3. Advertising interval no configurado
- **Fix**: Agregado `pAdv->setMaxPreferred(0x12)` junto al `setMinPreferred(0x06)` existente.

#### 4. MTU por defecto = 23 bytes → payload de 26 bytes truncado a 20
- **Causa**: El struct `ImpactPayload` ocupa 26 bytes, pero el MTU por defecto del ESP32 es 23 (payload útil = 20). Las notificaciones BLE llegaban truncadas.
- **Fix**: `BLEDevice::setMTU(185)` en `iniciarBLE()`.

#### 5. Error de compilación: `ImpactPayload does not name a type`
- **Causa**: El Arduino IDE auto-genera prototipos de funciones después de los `#include`. El struct estaba definido después de los includes, pero el prototipo de `enviarEvento(const ImpactPayload& p)` se insertaba antes de la definición del struct.
- **Fix**: Mover la definición del struct al inicio del archivo, antes de los `#include`.

#### 6. Error de compilación: `std::string` → `String`
- **Causa**: En versiones nuevas de la librería BLE de ESP32, `BLECharacteristic::getValue()` devuelve `String` (Arduino) en lugar de `std::string`.
- **Fix**: Cambiar tipo a `String` y `.data()` → `.c_str()` en `TimeSyncCallback::onWrite`.

### App Android (`BLEService.js`)

#### 7. `sendTimeSync` usaba BigInt → crash silencioso
- **Causa**: `uint64LeToBase64` usaba `BigInt(ms)` y `0xFFn`. En versiones antiguas de Hermes (motor JS), BigInt no está soportado. Esto hacía que `sendTimeSync` lanzara una excepción, lo que impedía que `monitorCharacteristicForService` se registrara → la app nunca recibía notificaciones BLE.
- **Fix**: Reescribir `uint64LeToBase64` sin BigInt, usando dos `uint32` (parte alta y baja).

#### 8. `getBigUint64` en `parsePayload`
- **Causa**: `dv.getBigUint64(0, true)` también usa BigInt internamente.
- **Fix**: Reemplazar con `dv.getUint32(0, true) + dv.getUint32(4, true) * 0x100000000`.

#### 9. `sendTimeSync` fatal bloqueaba la suscripción
- **Causa**: Si `sendTimeSync` fallaba por cualquier razón, la excepción propagaba fuera de `connectAndSubscribe` y nunca se llegaba a `monitorCharacteristicForService`.
- **Fix**: Envolver `sendTimeSync` en `try/catch` no-fatal.

#### 10. Filtro de scan por UUID fallaba en algunos Android
- **Fix**: Cambiar `[SERVICE_UUID]` → `null` en `startDeviceScan`, filtrar por nombre de dispositivo.

#### 11. Manager BLE destruido → loop `Cannot start scanning operation`
- **Causa**: `destroy()` llama `manager.destroy()` pero la variable `manager` seguía apuntando al objeto destruido. Al reabrir la app y llamar `startScan`, usaba el manager destruido.
- **Fix**: Poner `manager = null` en `destroy()` y recrear con `new BleManager()` al inicio de `startScan` si es null.

#### 12. Conexiones simultáneas → `Device ? is already connected`
- **Causa**: El scan callback es asíncrono. Si el device aparece múltiples veces antes de que `stopDeviceScan` haga efecto, se lanzaban múltiples `connectAndSubscribe` en paralelo.
- **Fix**: Flag `isConnecting` que bloquea intentos paralelos. Se resetea en `onDisconnected` y en el `catch`.

#### 13. Payload truncado a 20 bytes (MTU) — lado app
- **Causa**: Aunque el ESP32 tenía `setMTU(185)`, la negociación de MTU no se completaba a tiempo o no se iniciaba desde la app.
- **Fix**: `device.connect({ requestMTU: 185 })` — la app pide explícitamente el MTU al conectar.

#### 14. Stale closure en `alertsEnabled`
- **Causa**: `onDataReceived` se creaba con `useCallback([alertsEnabled])` pero se pasaba a BLEService una sola vez. Si el usuario cambiaba `alertsEnabled` después, el callback guardado en BLEService usaba el valor viejo.
- **Fix**: `alertsEnabledRef = useRef(true)` que se actualiza con `useEffect([alertsEnabled])`. `onDataReceived` usa `alertsEnabledRef.current`.

## Bugs conocidos pendientes (próxima sesión)

1. **ESP32 ya encendido al abrir la app**: Si el ESP32 lleva tiempo encendido y ya tiene una conexión fantasma, la app a veces no lo detecta. Hay que apagar/encender el ESP o desconectar y reconectar manualmente.

2. **Cerrar y reabrir la app no reconecta automáticamente**: El `autoReconnect` funciona para desconexiones en caliente, pero al cerrar la app completamente y reabrirla, el scan a veces no encuentra el dispositivo porque el ESP32 no está en estado de advertising (cree que sigue conectado).
   - Posible fix: en el ESP32, agregar un timeout de inactividad que fuerce `startAdvertising()` si no hay mensajes del cliente por N segundos.
   - Posible fix en app: al iniciar, cancelar cualquier conexión anterior antes de escanear.

3. **`newArchEnabled: false`**: Se deshabilitó la nueva arquitectura de React Native como medida preventiva. En el futuro se debería probar habilitar con una versión de `react-native-ble-plx` que soporte completamente la nueva arquitectura.

```cpp
void setup() {

    // Reinicio de advertising cada 8s si no hay cliente

    }
    // ...
}
```

### `BLEService.js` — cambios relevantes

```js
// manager como let (puede ser null y recrearse)
let manager = new BleManager();
let isConnecting = false;

// connect con MTU explícito
const connected = await device.connect({ requestMTU: 185 });

// sendTimeSync no-fatal
try { await sendTimeSync(connected); } catch (e) { console.warn(...); }

// Guard contra conexiones simultáneas
if (isConnecting || connectedDevice) return;
isConnecting = true;

// destroy limpio
destroy() {
    this.disconnect();
    if (manager) { manager.destroy(); manager = null; }
}
```
